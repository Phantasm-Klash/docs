# gotouhou 开发文档总纲

gotouhou 是一个原创主题的联机 PVP 弹幕卡牌网游。项目借鉴 STG 弹幕游戏中的低速移动、判定点、擦弹、Bomb、Replay、背版、诱导、输入延迟等设计语言，但基础游戏不绑定东方 Project 的文本、美术、角色、音乐或专有设定。

东方相关内容只作为可替换主题包和镜像素材方案处理。基础游戏内称为“卡牌”，东方主题文本可把同一概念显示为“符卡”。所有玩家可见文本必须走 i18n，不允许硬编码在玩法逻辑中。东方文本、美术和音频替换后续通过 Steam 创意工坊 Mod 形式接入，并独立处理授权、二创规则和商业权益风险。

首选客户端技术栈为 Godot 4.7，目标是保持客户端体量小、2D 开发效率高、跨平台导出简单。若后续实测 Godot 在包体或弹幕性能上不满足目标，备选轻量客户端为 LÖVE2D 或 raylib；不采用 UE 这类重型引擎作为首发方案。

基础开源服务端采用 Nakama + Go Runtime + PostgreSQL 作为业务服务器核心，负责账号、会话、匹配、房间、库存、卡组、活动和结算入库。强实时弹幕竞技、PVP 和 Boss 战调整为独立 C++ Battle Server，使用 ECDHE + KCP/UDP + protobuf + ChaCha20-Poly1305；客户端只上传输入和卡牌/模式请求，服务端结算移动、弹幕、擦弹、命中、计分和基础掉落。

Steam 商业发行版为买断制网游，没有游戏内充值和内购。Steam 购买校验、Steam 登录、成就、创意工坊、Steam Inventory 掉落宝箱、开箱、可交易卡牌/物品、Steam 市场对接等内容属于闭源 Steam 适配层。开源服务器版本只包含账号注册、开房大厅、对战、规则设置、卡组保存等基础功能。

## 阶段目录

- `00_overview/`：项目定位、术语映射、技术栈、开源边界、i18n、素材授权、路线图。
- `01_core_stg_client/`：单机 STG 核心手感、低速、判定、擦弹、Replay 和性能约束。
- `02_networked_match/`：联机权威对局、服务器权威确定性帧同步复盘、输入同步、延迟补偿、快照、断线重连和反作弊。
- `03_spellcard_deck_pvp/`：卡牌分类、效果、卡组构筑、平衡指标和 PVP 公平性。目录名保留 spellcard 作为早期命名，游戏内基础文本使用“卡牌”。
- `04_server_database_economy/`：服务端架构、数据库、经济系统、掉落宝箱、奖励和活动。
- `05_content_assets_ui/`：开源素材管线、多主题素材、i18n UI、音频和可访问性。
- `06_testing_release_ops/`：本地环境、自动化测试、延迟测试、平衡模拟、部署和封测。
- `07_steam_closed_layer/`：Steam 闭源适配层、购买校验、成就、创意工坊、库存市场和商业服边界。
- `08_game_modes/`：考证评级、大逃杀、世界 Boss、副本 Boss 和模式共享服务端接口。

## 开发顺序

最早的顺序是从文档、单机、联机、卡牌、经济、内容到 Steam 层逐步推进。当前代码已经并行完成了多个 MVP 切片，因此后续顺序调整为围绕“服务器权威闭环”收敛：

1. 维护文档和规则规格，所有新实现继续回写到对应阶段文档。
2. 收尾 Godot 本地 STG 核心：手感、Replay、练习关卡、弹幕数学基底、机体、自机模式和性能验收。
3. 冻结 v0.1 共享规则/配置契约：关卡、弹幕、卡牌、机体、模式、奖励和 Replay 元数据必须能被客户端展示、服务端权威执行、测试复现。
4. 推进 Gensoulkyo 权威对局：先稳定 HTTP MVP 和 live check，再冻结 60Hz v0.1 tick、定点单位、输入窗口、canonical state hash 和 golden replay fixture；随后迁移业务层到 Nakama RPC/WSS，并把强实时战斗迁移到 C++ Battle Server、KCP、protobuf、ECDHE 和 ChaCha20-Poly1305。
5. 将当前内存经济迁移到 PostgreSQL：账号、库存、卡组、宝箱、保底、奖励、活动、排行榜、Replay audit 和世界 Boss 状态都要可重启、可审计。
6. 完成卡牌/卡组首批内容和平衡：至少 30 张基础卡牌、排位限制、强干扰限制、Boss/大逃杀专用规则和自动平衡报告。
7. 将调试 UI 升级为正式 Godot 界面：练习、关卡、卡组、宝箱、匹配、在线对局、模式、Replay、设置和结果流程都要键盘/手柄可闭环。
8. 完成测试和运维门槛：CI、Linux 构建、Godot headless、Go tests、延迟矩阵、Replay 校验、异常对局审计、资产授权和部署监控。
9. 在开源 MVP 可自托管、可持久化、可复现后，再进入赛季 0 封测。
10. 闭源 Steam 适配层最后接入购买校验、成就、Workshop、Inventory 和市场，且不改变开源仓库边界。

## 全局原则

- 客户端不可被信任，不提交分数、擦弹数、开箱结果或权威位置。
- 随机弹幕、宝箱掉落、开箱、活动奖励全部由服务端或 Steam 受信接口生成和记录。
- Replay 保存输入流、随机种子、卡组快照和版本号，不保存视频。
- v0.1 联机战斗采用服务器权威确定性帧同步：客户端只上传输入和模式动作意图，C++ Battle Server 负责固定 tick、快照、事件、Replay 和签名结算。
- PVP 中视觉干扰不得遮挡判定核心信息。
- 所有素材必须记录授权来源，优先 CC0、MIT、Apache-2.0、CC-BY 等可商用协议。
- 所有可见文本使用 i18n key；基础包、东方主题包和其他主题包只替换资源，不改玩法逻辑。
- GitHub 开源仓库不包含 Steamworks SDK、Steam Web API Key、商业服密钥、市场掉落策略和闭源风控逻辑。
- 多模式共享服务器权威框架；模式之间的卡池、卡牌效果、奖励和排行榜可独立配置。
- Nakama 不取缔，继续作为业务服务器核心；C++ Battle Server 只接管高频战斗模拟和战斗网络热路径。
