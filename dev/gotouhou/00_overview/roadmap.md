# 路线图

状态更新时间：2026-06-26

## 当前阶段判断

按最早的 1-8 阶段表，项目已经不在“仓库初始化/最小客户端原型”阶段。结合 `SpellKard` 和 `Gensoulkyo` 的现有代码，当前真实状态是：

- 阶段 1 已完成：文档目录、边界、玩法与技术规格已经建立。
- 阶段 2 已基本成型：Godot 本地 STG 厚原型具备移动、低速、判定、擦弹、Bomb、计分、Replay、练习关卡、机体、12 类弹幕数学基底和自动化烟测。
- 当前主线应视为阶段 3：联机权威对局和服务端拆分收敛期。`Gensoulkyo` 已有 Go 标准库 HTTP 内存 MVP，`SpellKard` 已有 HTTP 契约适配和 Godot `HTTPRequest` 包装；接下来保留 Nakama 作为业务服务器核心，并把强实时 PVP/Boss 战斗迁移到 C++ Battle Server、KCP、protobuf、ECDHE 和 ChaCha20-Poly1305。
- 阶段 4、阶段 5、阶段 6、阶段 8 已有 MVP 切片并行推进：卡组/卡牌、服务端经济、i18n/主题/测试、多模式 Boss/考证/大逃杀表面均已启动。
- 阶段 7 未开始，仍应等开源 MVP 稳定、资产授权和商业边界再次复核后再进入。

因此后续计划不再按完全线性方式“做完一个阶段才开下一个阶段”，而是围绕服务器权威闭环收敛：先统一规则和协议，再做持久化与实时传输，随后补正式 UI、内容、运维和 Steam 闭源层。

## 阶段 1：文档与原型规范

目标：建立完整文档结构，并把 STG 核心概念转成可实现规格。

实际状态：已完成，后续只随实现变化增量维护。

验收：

- `dev/docs/gotouhou/` 下所有阶段目录存在。
- 低速、判定点、擦弹、Replay、延迟补偿、卡牌、i18n、开源边界、数据库均有独立文档。
- v1 范围和暂不做内容明确。

## 阶段 2：单机 STG 核心

目标：在 Godot 中完成可玩的本地弹幕原型。

实际状态：基本成型，处于手感、视觉和性能验收收尾。`SpellKard` 已实现固定 tick 移动、低速、判定点、擦弹、命中、Bomb/deathbomb、射击、计分、Replay 保存/读取/播放、练习关卡、Pattern Lab、4 个机体和 12 类弹幕基底。

验收：

- 高速/低速移动、判定点、擦弹、命中、Bomb 可用。
- 至少 8 种弹幕模式。
- 本地 Replay 可保存和重放。
- 同屏弹幕上限内稳定 60 FPS。

剩余工作：

- 用最终或近最终视觉资源复核 60 FPS 和可读性。
- 把临时程序化 UI 覆盖层收敛为正式练习、关卡选择、回放和设置界面。
- 继续用平衡模拟、延迟矩阵和烟测保护本地玩法回归。

## 阶段 3：联机权威对局

目标：完成服务器权威 1v1 对局。实现路径调整为：先稳定当前 HTTP MVP，再迁移业务层到 Nakama Go Runtime / HTTPS RPC / WSS，并将高频战斗层迁移到 C++ Battle Server / KCP / protobuf。同步路线定义为“服务器权威确定性帧同步”：客户端只提交输入和模式动作意图，C++ 战斗服固定 tick 推进并输出快照、事件、Replay 和签名结果。

实际状态：进行中，是当前主线。`Gensoulkyo` 已实现匿名登录、bootstrap、匹配/房间/ready、tick 输入、服务器弹幕快照、服务端 active-card 快照、事件游标、断线重连、结算、Rematch 和 Replay audit 的内存版 HTTP MVP；`SpellKard` 已能消费这些契约并把服务器权威结果投射到客户端模型。当前 Go MVP 已使用 60Hz tick，客户端网络模型已有 2-4 tick 输入延迟、快照修正和 KCP/protobuf/AEAD 状态脚手架；`PhK-Protocol` 已有 battle input/snapshot/result proto 草案；`PhK-BattleServer` 已有 packet/ticket/dispatcher 骨架。下一步不是取缔 Nakama，而是把账号/匹配/房间/业务 WSS/存储迁入 Nakama + Go Runtime，同时新建 C++ Battle Server 承担弹幕战斗热路径。

验收：

- 客户端按 tick 上传输入。
- 服务器结算位置、弹幕、擦弹、命中和分数。
- 80ms 网络延迟下低速手感可接受。
- 断线重连和对局 Replay 可用。
- 同一 seed、ruleset、deck snapshot 和输入流，在 C++ authoritative simulation 中生成稳定 state hash；客户端 replay 展示不得改变权威事件和结算。

剩余工作：

- 新建或规划 `PhK-Protocol`，抽取 protobuf、业务 envelope、battle ticket、ruleset schema、错误码和代码生成。
- 冻结 v0.1 tick rate 为 60Hz，明确输入窗口、定点单位、snapshot 频率、state hash canonical 格式和 golden replay fixture。
- 将 HTTP MVP 的业务能力迁移为 Nakama RPC + 业务 WSS，保留当前 HTTP 集成测试作为契约回归和 fallback。
- 新建 C++ Battle Server，并迁移 PVP/Boss tick、弹幕、碰撞、卡牌生效、快照、事件、Replay 和战斗结果签名。
- 增加真实网络条件下的 30/80/150/250ms 验收，覆盖 KCP、丢包、重连、快照修正和 Replay 复现。
- 完成正式在线匹配、房间、准备、对局、结算和重赛界面。

## 阶段 4：卡牌与卡组

目标：完成卡组构筑、出牌、反制、主题文本替换和基础平衡记录。

实际状态：进行中。客户端已有卡牌目录、20 张卡组校验、4 张起手、手牌/能量/冷却、自身强化、弹幕修改、护盾、抽牌/能量效果、卡组编辑、服务器库存/升级投射；服务端已校验卡组保存、卡牌拥有、升级成本、出牌槽位、能量和冷却。

验收：

- 20 张卡组、起手 4 张、手牌上限 6。
- 至少 30 张卡牌覆盖主要类型。
- 基础文本显示为“卡牌”，东方主题文本可显示为“符卡”。
- 排位卡池和强干扰限制可配置。

剩余工作：

- 定稿首批公开卡池和排位卡池，补足至少 30 张可平衡卡牌。
- 把卡牌效果从客户端本地调试逻辑收敛到服务端权威规则，并用版本号随 bootstrap 下发。
- 为强干扰、共享场、Boss/大逃杀专用卡池建立独立平衡表和禁用规则。

## 阶段 5：服务端经济

目标：完成账号资产、背包、掉落宝箱、奖励、活动和排行榜。

实际状态：进行中。`Gensoulkyo` 内存 MVP 已拥有服务端库存、卡组、卡牌升级、宝箱池、开箱、保底、奖励、任务、活动、排行榜投射、幂等结算和客户端伪造结果拒绝；`SpellKard` 已能显示并消费这些服务器形状的数据。

验收：

- 宝箱掉落和开箱结果由服务端生成并落库。
- 开源版使用非市场化掉落，Steam 商业版使用 Steam Inventory 物品。
- 重复卡、分解或兑换、卡组保存可用。
- 对局奖励幂等，重复领奖无效。

剩余工作：

- 实现 PostgreSQL migrations，并把账号、库存、卡组、宝箱、奖励、活动、排行榜、Replay audit 和世界 Boss 状态从内存迁移到持久化存储。
- 补管理/运维接口、数据修复脚本和审计日志。
- 明确开源经济与 Steam Inventory 闭源经济的转换边界。

## 阶段 6：内容、测试与上线

目标：完成素材管线、UI、延迟测试、封测和监控。

实际状态：进行中。客户端已有资产 manifest、主题 manifest、i18n、可访问性、输入配置、音频设置模型、平衡模拟、延迟矩阵、Godot headless 检查和 Gensoulkyo live HTTP 检查；生产素材、正式 UI、音频播放、CI 和部署监控仍未完成。

验收：

- 素材授权清单完整。
- i18n 文本覆盖基础主题，主题包替换流程可用。
- 关键 UI 流程闭环。
- 压测、Replay 校验、异常对局审计通过。
- 可开放赛季 0 封闭测试。

剩余工作：

- 建立 CI：Godot headless、Go tests、license manifest、文档链接和格式检查。
- 用正式 UI 场景替换调试覆盖层，保证键盘/手柄可完整完成登录、匹配、对局、卡组、宝箱、回放和设置流程。
- 补生产可用美术、音频、字体和授权记录，再做封闭测试计划。

## 阶段 7：Steam 闭源商业层

目标：完成 Steam 买断发行、购买校验、成就、Workshop、Inventory、掉落宝箱和市场交易接入。

实际状态：未开始。该阶段不应阻塞当前开源 MVP，但任何设计都必须继续遵守“Steam SDK、密钥、商业服风控和市场掉落策略不进入开源仓库”的边界。

验收：

- Steam 登录和 App 所有权验证可用。
- 未购买玩家无法登录官方商业服。
- 成就和统计可同步到 Steam。
- Workshop 主题包可订阅、下载、启用和禁用。
- 宝箱和卡牌可通过 Steam Inventory 发放，并按配置决定是否可交易或可上市场。
- Steam 相关密钥和风控逻辑不进入 GitHub 开源仓库。

## 阶段 8：多模式服务端

目标：在服务器权威框架上实现考证评级、大逃杀、世界 Boss 和副本 Boss。

实际状态：进行中。客户端已有考证评级、大逃杀候选选择、Boss 站位/让渡/结果投射；服务端已有考证资料投射、评级结算、Boss HP/伤害/世界 Boss 每日次数/持久血量/击败公告、副本 Boss 必须本局击败才通关，以及模式动作校验。

验收：

- 考证模式可授予评级，同评级优先匹配，rank 前 30% 可参加下一阶段评级。
- 大逃杀支持 5-10 人开局、每人 4 张入公共卡池、每回合 30 秒 3 选 1、无间隔连续回合。
- 大逃杀卡牌效果与其他模式独立，并按第零回合排名计算延迟生效顺序。
- 世界 Boss 支持每日挑战次数、4-8 人站位、卡牌让渡、服务器持久血量扣除和血量为 0 世界通告。
- 副本 Boss 使用同类组队规则，但必须本局击败才算通过。
- 所有模式 Replay、结算、奖励和反作弊规则可复现。

剩余工作：

- 完成大逃杀连续回合、公共卡池、候选生成、零回合排序和奖励结算。
- 完成考证资格链路、评级赛季、榜单前 30% 资格和匹配优先级的持久化。
- 完成世界 Boss 和副本 Boss 的正式机制、星级条件、奖励、排行榜和 Replay 审计。

## 调整后的近阶段计划

1. 协议收敛：冻结 v0.1 `PhK-Protocol`，统一 protobuf、业务 envelope、battle ticket、错误码、ruleset schema 和生成物版本。
2. 业务层迁移：保留当前 HTTP MVP 作为契约测试，把账号、资料、饰品、道具、库存、卡组、宝箱、活动、排行榜、匹配、房间和业务 WSS 迁入 Nakama + Go Runtime。
3. 战斗层迁移：新建 C++ Battle Server，使用 ECDHE + KCP/UDP + protobuf + ChaCha20-Poly1305，逐步接管 PVP、Boss、大逃杀等高频 tick。
4. 服务端持久化：实现 PostgreSQL migrations，把账号、库存、卡组、宝箱、奖励、活动、排行榜、Replay audit、battle ticket、battle allocation 和世界 Boss 状态迁移为可重启、可审计、可回滚的数据模型。
5. 客户端正式 UX：把练习、关卡、卡组、宝箱、匹配、在线对局、模式、回放、设置和结果界面从调试覆盖层升级为可交付界面，并补业务 WSS/KCP 战斗客户端。
6. 内容和平衡：补足首批 30+ 卡牌、4+ 机体、公开练习关卡、Boss/大逃杀专用卡池，并让平衡模拟成为合并门槛。
7. 多仓库构建：建立 `PhK-Protocol`、`Gensoulkyo`、`PhK-BattleServer`、`SpellKard` 的版本锁定、CI、远端 Linux 构建、Docker artifact、延迟矩阵、Replay 复现和 license manifest。
8. 封测准备：只有在开源 MVP 可自托管、可重启、可复现、可审计后，才进入赛季 0 封测和 Steam 闭源层接入。
